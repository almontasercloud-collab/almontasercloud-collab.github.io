## ICMP
The **Internet Control Message Protocol (ICMP)** is the protocol behind the famous Ping command, making it one of the most recognizable tools in networking. Despite being designed for diagnostics and error reporting, ICMPipe demonstrates that it can be pushed much further.

**ICMPipe** is nothing more than an experiment that attempts to transfer data through the payload bytes of ICMP packets using a custom transfer algorithm. It demonstrates how an algorithm can provide data exchange to a protocol originally intended for diagnostics.

## ICMPipe algorithm

The ICMPipe transfer algorithm consists of two main phases. In the first phase, the client requests a file by sending its absolute path to the server. If the file exists, the server responds with a File Acknowledgment (FA) containing the file size. Otherwise, the transfer is terminated. The second phase begins only after a successful acknowledgment, where the client requests the file and the server streams it back as a sequence of data chunks carried in ICMP requests payloads.

>**Flags:** **FR** = File request, **FA** = File Acknowledgment, **FP** = File pull, **FD** = File data
{: .prompt-tip }


```bash

Phase 1:

Request(Client): Initiate file request (FR:absolute path)
Reply(Server): Initiate file request (FR:absolute path) <- Default

Request(Server): Size response (FA:size:FA) OR (FA:File Not Found:FA)
Reply(Client): Size response (FA:size:FA) OR (FA:File Not Found:FA) <- Default

Proceed based on "IF FOUND" condition on client side.

Phase 2:

Request(Client): Initiate file pull (FP:absolute path). 
Reply(Server): Initiate file pull (FP:absolute path). <- Default

Request(Server): Transmit first data chunk (FD:Data chunk). 
Reply(Client): Acknowledge first data chunk (FD:Data chunk). <- Default

Request(Server): Transmit next data chunk (FD:Data chunk). 
Reply(Client): Acknowledge next data chunk (FD:Data chunk). <- Default

...
Terminate when total received size >= FA file size
Reassemble received payloads

```

The algorithm may appear logically complete at first glance, but in practice it remains minimal and does not address several critical reliability scenarios:

- How to handle a lost or dropped FD (File Data) packet during transmission, including detection and retransmission logic.

- How to guarantee correct ordering of chunks when packets arrive out of sequence under real network conditions.

- How to detect corruption or tampering in payload data beyond basic size tracking.

- How to handle partial session recovery if the connection is interrupted mid-transfer.

- How to scale the mechanism under congestion, rate limiting, or ICMP filtering by intermediate network devices.

These limitations and many more, including edge cases found in real-world network conditions, exist because ICMPipe is designed as a minimal proof-of-concept, not a full transport protocol. The goal is to demonstrate that structured data transfer is possible over ICMP payloads using a custom algorithm.

## ICMPipe in Action

> ICMPipe depends on standard ICMP echo request/reply behavior to function as a transport layer. In rare cases where this behavior is altered or not supported, specific code sections must be uncommented and the project rebuilt to adapt to the environment.
{: .prompt-info }
> Source code, executables and installation steps are available on [**`This Github Repo`**](https://github.com/almontasercloud-collab/ICMPipe)
{: .prompt-tip }

In this section, ICMPipe is demonstrated in a controlled lab environment to validate its file transfer behavior over ICMP payloads. A **Linux-based** ICMPipe client (172.16.2.11) is used to request and retrieve a text file from a **Windows** host acting as the ICMPipe server (172.16.2.102). During the transfer, ICMP traffic is captured and analyzed to observe the FR/FA/FP/FD exchange pattern, payload encoding, and the request–reply behavior at the packet level.

The captured traffic is then used to verify correct sequencing and reassembly of the transmitted data on the client side. This provides a clear view of how the custom algorithm operates over standard ICMP echo request/reply communication.

In the next stage, the server implementation is modified to increase the maximum ICMP payload size, followed by a full rebuild of the system. This adjustment is introduced to evaluate how the algorithm behaves under larger data loads and to expand the effective throughput of each ICMP packet.

Finally, the updated setup is tested using an image file instead of a text file. This second experiment highlights how ICMPipe handles binary data transfer, larger payload segmentation, and increased reassembly complexity on the client side, while still preserving the same FR/FA/FP/FD protocol structure.

- Decoded payload data:

> In the first ICMP Echo Request, ICMPipe client prepends the FR flag to the absolute path of test.txt, signaling the server to locate and transmit the requested file.

#### The second ICMP echo request packet: **File Ack**

![server flags](/assets/img/posts_photos/icmpipe/txt_second_packet.png)

> The second ICMP echo is originated from the server machine at (172.16.2.102) and sent to (172.16.2.11) the client.

- The payload of the second ICMP Echo Request packet, Base64-encoded:

```bash
RkE4NjhGQWFiY2RlZmdoaWprbG1ub3BxcnN0dXZ3YWJjZGVmZw==
```
- Decoded payload data:

```bash
FA868FAabcdefghijklmnopqrstuvwabcdefg
```

> In the second ICMP Echo Request, the ICMPipe server constructs the payload using the format **`FA[file size]FA`**, followed by the default Windows ICMP Echo Request payload (abcdefghijklmnopqrstuvwxyz...) to occupy the remaining bytes. Note that it will use **`FA["File Not Found"]FA`** if ICMPipe server didn't find the requested file.

#### The third ICMP echo request packet: **File pull**

![server flags](/assets/img/posts_photos/icmpipe/txt_third_packet.png)

> The first ICMP echo is originated from the client machine at (172.16.2.11) and sent to (172.16.2.102) The server.

- The payload of the second ICMP Echo Request packet, Base64-encoded:

```bash
RlBDOlxVc2Vyc1xBZG1pbmlzdHJhdG9yXERlc2t0b3BcdGVzdC50eHQ=
```
- Decoded payload data:

```bash
FPC:\Users\Administrator\Desktop\test.txt
```

> In the third ICMP Echo Request, the client constructs the payload using the format **`FP[file path]`**, signaling the server to begin transmitting the contents of the requested file.

#### The remaining ICMP echo requests packets: **Data chunks**

![server flags](/assets/img/posts_photos/icmpipe/data_chunks.png)

> All the remaining ICMP echo requests are originated from the server machine at (172.16.2.102) and sent to (172.16.2.11) the client.

- The payload of the data chunks ICMP Echo Requests packet, **FD** + Base64-encoded data chunk:

```bash
FDSUNNUGlwZSBUZXN0IEZpbGUNCj09PT
```
- Decoded payload data after removing the **FD** flag:

```bash
ICMPipe Test File
===
```
> In the final phase, the server transmits the file as a sequence of ICMP Echo Request packets. Each packet carries a payload in the format **`FD[Base64-encoded data chunk]`**, where the file data is Base64-encoded before transmission. Upon receiving each packet, the client removes the FD flag, decodes the Base64 payload, and appends the recovered data to the destination file until the transfer is complete.

## Conclution

ICMPipe demonstrates a simple implementation of file transfer over ICMP Echo messages using a lightweight application-layer protocol. Throughout this article, we examined the complete transfer lifecycle—from the initial file to chunk transmission and final file reconstruction.

While ICMP was not designed for general-purpose file transfer, ICMPipe demonstrates how custom protocol logic can coordinate communication over ICMP for experimental and educational purposes. It also provides a practical example of building a simple application-layer protocol on top of an existing network protocol.
